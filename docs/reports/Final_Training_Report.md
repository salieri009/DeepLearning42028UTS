# Final Training Report — CrowdNav YOLOv8m on JRDB

*Last updated: 2026-05-02*

## 1. Executive Summary

We trained a single-class **person detector** for the CrowdNav accessibility-navigation system on the JRDB dataset (Stanford robot egocentric video). The pipeline progressed through three training phases on a local RTX 3050 Laptop GPU (4 GB VRAM), each phase building on the weights of the previous one. The final model reaches **mAP@0.5 = 0.4475** and **mAP@0.5:0.95 = 0.2086** on the held-out validation split, a **+75 % improvement over the pseudo-label baseline** at the same hardware budget. Cloud scale-up on AWS SageMaker (`ml.g4dn.xlarge`, T4 16 GB) is the planned next step to push accuracy further by training a larger backbone with a higher batch size for the full 50-epoch schedule the local laptop cannot accommodate.

**Integration note (backend / inference, no frontend):** When training completes, **`best.pt`** is exported to **ONNX** and loaded by **Spring `crowdnav-api`** (ONNX Runtime). See [`docs/runbooks/post_train_spring_onnx.md`](../runbooks/post_train_spring_onnx.md) and [`docs/TechSpec.md`](../TechSpec.md) §4.2 / §6.

## 2. Data Pipeline

### 2.1 Source dataset
- **JRDB** (Stanford, 2019/2024): 27 train+val sequences captured from a 360° multi-camera robot rig.
- Cameras used: `image_0` (front) and `image_2` (right ~72°). Native frame size 752 × 480 RGB.
- Official 2D annotations (`labels_2d`) provide pixel-space bounding boxes plus consistent tracking IDs (`pedestrian:N`).

### 2.2 JRDB → YOLO conversion
A new utility, **`scripts/jrdb_train_to_yolo.py`**, parses each per-camera JSON, normalises pixel-space boxes to YOLO format, and writes one `.txt` per frame per camera. Boxes flagged `attributes.no_eval = true` (severely occluded / annotator-uncertain) are excluded. The conversion summary on the training set:

| Camera | Sequences | Label files | Boxes |
|---|---|---|---|
| image_0 | 27 | 27,027 | 266,927 |
| image_2 | 27 | 24,341 | 144,410 |
| **Total** | **27** | **51,368** | **411,337** |

Skipped boxes: 48,421 with `no_eval=true` (~10 %).

### 2.3 Sequence-level split
Splitting was performed at the sequence level (seed 42, 80/10/10) using `src.data.prepare.split.run()`, separately per camera so that a single frame contributes one YOLO label per camera view.

| Split | image_0 | image_2 | **Total** |
|---|---|---|---|
| train | 20,636 | 18,957 | **39,593** |
| val | 1,369 | 1,125 | **2,494** |
| test | 4,975 | 4,217 | **9,192** |

A path bug in `dataset_config.write_data_yaml` (the original `path: "."` is interpreted as the current working directory by Ultralytics, not the YAML file location) was fixed to write the absolute resolved path of `output_dir`.

## 3. Training Phases

All three phases share the same architecture, optimiser, augmentation, and image size. They differ only in **the source of the labels** and **how many epochs were run**.

### 3.1 Common configuration

| Component | Setting |
|---|---|
| Architecture | YOLOv8m (Ultralytics) — 170 layers, 25.86 M params, 79.1 GFLOPs |
| Pretrained | MS COCO |
| Output classes | 1 (`person`) |
| Image size | 640 × 640 |
| Optimiser | AdamW (auto-selected) |
| Initial LR / Final LR | 6.66 × 10⁻⁴ → 1.72 × 10⁻⁴ (cosine decay) |
| Weight decay | 5.0 × 10⁻⁴ |
| Augmentation | Mosaic 1.0, fliplr 0.5, HSV (0.015 / 0.7 / 0.4), translate 0.1, scale 0.5, randaugment, erasing 0.4 |
| AMP | enabled |
| Hardware | RTX 3050 Laptop (4 GB VRAM), CUDA 12.1, PyTorch 2.5.1 |

### 3.2 Phase A — Pseudo-label baseline (`crowdnav_yolo` cycle 0)
- Labels: generated by the COCO-pretrained YOLOv8m itself, with TTA (`augment=True`) and `model.track()` for temporal consistency.
- Hyperparameters: 13 epochs, batch 4, workers 4.
- **Best epoch: 4 / 13** — mAP@0.5 = 0.256, P = 0.371, R = 0.373.
- Validation loss starts rising from epoch 5; clearly noise-propagation overfitting.
- Bounding-box density per image: **3.20** (vs 7.91 for ground-truth) — confirming that the self-labelled pipeline misses ~60 % of pedestrians, especially in heavily occluded crowds.

### 3.3 Phase B — GT fine-tune, batch 8 (`crowdnav_yolo` GT 1st run)
- Labels: JRDB official `labels_2d` after the YOLO conversion described in §2.2.
- Hyperparameters: 10 epochs, batch 8, workers 6.
- **Final epoch: 10 / 10** — mAP@0.5 = 0.380, mAP@0.5:0.95 = 0.188, P = 0.512, R = 0.396.
- Throughout the 10-epoch run the curves still trend upward (mAP@0.5: 0.361 → 0.380), indicating the model has not yet plateaued on this hardware.

### 3.4 Phase C — GT fine-tune continuation, batch 8 (`crowdnav_yolo` GT 2nd run)
- Resumed from the Phase B `best.pt`, with `lr0 = 0.001` for fine-tuning.
- Hyperparameters: 20 epochs, batch 8, workers 6.
- Wall-clock: **12.158 hours**.
- Mosaic augmentation closes at epoch 10, after which the training loss drops noticeably and validation precision keeps climbing.
- **Best epoch: 18 / 20** — mAP@0.5 = 0.4475, mAP@0.5:0.95 = 0.2086, P = 0.6684, R = 0.4302.

## 5. Analysis

1. **Pseudo-label is a useful warm-start, not an end state.** The Phase A model saturates around mAP@0.5 = 0.26 because every iteration trains the network on its own (noisy) outputs. Switching to JRDB official annotations alone delivered the largest single gain in the project (+48 % at Phase B end).
2. **Continuation training extracts further accuracy without bigger hardware.** Phase C added 20 epochs from the Phase B best.pt and lifted mAP@0.5 from 0.380 → 0.4475 (+18 %) — driven mainly by precision (+30 %) once mosaic augmentation closed at epoch 10 and the model could specialise on confirmed boxes.
3. **The validation set is small and sequence-correlated.** Only two sequences (~2,500 frames) form val, so single-epoch fluctuations of ±0.02 mAP are noise. Best-epoch selection via fitness, not last-epoch metrics, is essential.
4. **mAP@0.5:0.95 lags behind mAP@0.5 (0.21 vs 0.45).** This indicates good classification but loose box localisation, especially on small/distant or partially-occluded pedestrians — the exact targets that matter for crowd-density estimation. A larger backbone (yolov8l/x) and longer schedule should improve box quality.
5. **Hardware was the binding constraint.** RTX 3050 Laptop forced batch 8 and rate-limited each epoch to ~33 minutes. SageMaker `ml.g4dn.xlarge` (T4 16 GB) doubles VRAM and supports batch 32 at the same image size, halving effective wall-clock per epoch and unlocking a 50-epoch schedule.

## 6. Next Steps — Cloud Scale-up

The migration plan and full command reference are recorded in [`SageMaker_Migration_Plan.md`](../runbooks/SageMaker_Migration_Plan.md).

Recommended SageMaker baseline:

```bash
yolo train \
  model=yolov8m.pt \
  data=data/processed/splits/data.yaml \
  epochs=50 batch=32 imgsz=640 workers=4 \
  patience=20 project=runs/train name=crowdnav_sagemaker
```

Expected lift: mAP@0.5 ≈ 0.55 (+22 %) at ≈ $4 per run on `ml.g4dn.xlarge`.

If headroom permits, the higher-capacity option is yolov8l with batch 16; this should reach mAP@0.5 ≈ 0.60 but doubles wall-clock to ~11 hours.

## 7. Reproducibility

```bash
# 1. JRDB official labels → YOLO format
python scripts/jrdb_train_to_yolo.py \
  --src-dir <path-to>/jrdb_train_labels/labels/labels_2d \
  --out-dir data/processed/labels_gt \
  --cameras image0 image2 --img-w 752 --img-h 480 --exclude-no-eval

# 2. Sequence-level split (per camera)
python -c "
from pathlib import Path
import sys; sys.path.insert(0, '.')
from src.data.prepare import split as split_api
for cam, prefix in [('image_0', 'image0'), ('image_2', 'image2')]:
    split_api.run(
        src_labels=Path(f'data/processed/labels_gt/{cam}'),
        src_images=Path(f'data/raw/images/{cam}'),
        output_dir=Path('data/processed/splits'),
        stem_prefix=prefix,
    )
"

# 3. Local training (Phase B)
python scripts/train_yolo.py \
  --model-cfg yolov8m.pt \
  --data-yaml data/processed/splits/data.yaml \
  --epochs 10 --batch 8 --workers 6 --imgsz 640 --validate

# 4. Continuation (Phase C)
python scripts/train_yolo.py \
  --model-cfg runs/train/crowdnav_yolo/weights/best.pt \
  --epochs 20 --batch 8 --workers 6 --validate
```
