# Data layout and DVC

- **Dataset root:** `<repository-root>/data/` (not under `train/`). Raw frames, processed labels, and YOLO splits live here; paths match `.gitignore` (large binaries via DVC when configured).
- **Training defaults:** CLI tools under `train/` resolve `data/...` against the repository root (see `train/src/repo_paths.py`), so you can `cd train` and run `python scripts/train_yolo.py` without passing `--data-yaml` when splits are at `../data/processed/splits/` from the repo perspective — i.e. `data/processed/splits/data.yaml` at repo root.
- **`data.yaml`:** Keep `path: .` relative to the directory that contains `train/`, `val/`, and `test/` (usually `data/processed/splits/`), as generated by the split step.
- **DVC:** Configuration lives in `<repository-root>/.dvc/`. Use `dvc pull` / `dvc push` from the repository root.

## SageMaker Notebook — minimal sync (not full repo clone)

The repository can be large; for **SageMaker Notebook / Studio on EBS**, sync only what training needs:

- **Required:** `data/processed/splits/` (images, YOLO labels, `data.yaml`) at the same paths as locally.
- **Required:** `train/` — at least `requirements.txt`, `scripts/`, `src/` (including `src/training/`, `src/repo_paths.py`), and optionally `notebooks/`.
- **Optional at bundle root:** top-level `requirements.txt` if it only pulls `-r train/requirements.txt`.
- **Usually omit:** `application/`, most of `docs/`, `infra/` (unless launching managed jobs from the instance), `.git` artifacts, caches.

Keep the same directory shape as the full repo (`<root>/data/...` and `<root>/train/...`) so `train/src/repo_paths.py` and `python scripts/train_yolo.py` behave identically after `cd train`.
